# PRD: KDS（Kitchen Display System）— 土鍋飯ぽかぽか（沖縄）

## エグゼクティブサマリー

- **背景:** 土鍋飯ぽかぽか（沖縄）は台湾人観光客を主客層とし、ネパール人スタッフと日本人スタッフが混在する特殊な厨房環境を持つ。口頭・紙ベースのオーダー伝達は言語障壁・見落としリスクが高く、提供速度の改善余地が大きい。KDS市場はCAGR 7.78%で成長中であり、現時点が参入の最適ウィンドウとCMOが判断。CCC取締役会はMVPとしてレイアウトA（カードグリッド）＋3言語対応の即時実装を条件付き承認した。
- **プロダクトビジョン:** 多言語スタッフが厨房内でオーダーを一覧把握し、提供ミスと提供遅延をゼロに近づけるオフライン完結型KDSをブラウザ単一ファイルで実現する。
- **最終判断:** conditional — MVPスコープはレイアウトA（カードグリッド）のみ、3言語対応（日本語・ネパール語・繁体字中国語）をMVPに含める、Cレーン分割はv1バックログへ。
- **PRD作成日時:** 2026-05-23T00:00:00+09:00
- **参照レポート:** CCC取締役会議レポート KDS（Kitchen Display System）画面設計・実装 2026-05-23

## ターゲットユーザー

CMOレポートの顧客セグメント分析に基づき、KDSの直接利用者を以下の2ペルソナに定義する。台湾人観光客はKDSの間接受益者であり、直接ユーザーには含めない。

### ペルソナ

| ペルソナ名 | 属性 | 課題 | 期待する価値 |
| --- | --- | --- | --- |
| ビマル（ネパール人スタッフ） | 20代・厨房補助スタッフ・来日2年・日本語読解が限定的 | 口頭指示や紙票の漢字が読めず、どの卓の何を作ればよいか混乱する。日本語メニュー名を見ても品目が一致しない | 色・アイコン・ネパール語表示で品目と卓番号を即座に把握できる。日本語が読めなくても自律的に作業できる |
| 田中さん（日本人スタッフ） | 30代・ホールリーダー・厨房サポート兼任 | ホールとの往復でオーダー状況を随時把握できない。どの卓が待ち状態か・何分経過しているか瞬時に分からない | テキスト重視の日本語表示で卓番号・経過時間・残り品目を一覧確認できる。ホールに出る前に厨房状況を10秒で把握できる |

## 解決する課題

CMOが特定した市場ニーズおよびヒアリング課題を整理する。

1. **言語障壁によるオーダーミス:** ネパール人スタッフが日本語票を誤読し、品目や数量を間違える。
2. **提供遅延の不可視化:** どの卓が何分待っているか厨房から見えず、クレームが発生してから気づく。
3. **口頭伝達の不確実性:** ホールから厨房へのオーダー伝達が口頭に依存し、ピーク時（最大15組）にロストオーダーが発生する。
4. **完成確認の属人化:** 品目完成のチェックが個人の記憶に依存し、同じ料理を二度作る・作り忘れるケースがある。
5. **Wi-Fi不安定環境でのシステム停止リスク:** クラウドPOSと連携するシステムはWi-Fi断で停止するため、厨房端末の自律動作が必要。

## プロダクト概要

土鍋飯ぽかぽか向けKDSは、POSシステムから`window.KDS_ORDERS`配列へ書き込まれたオーダーデータを1秒ごとに読み取り、厨房タブレット（Android × 2台）に横並びカードグリッドとして表示するシングルHTMLアプリケーションである。

外部CDN・ライブラリ依存ゼロ、単一HTMLファイルで完結し、Wi-Fi断・ネット断時もオフライン動作する。複数タブレット間のリアルタイム同期はBroadcastChannel APIで実現する。UI言語は日本語・ネパール語・繁体字中国語の3言語を切り替え可能とし、言語設定はLocalStorageに永続化する。

CTOの推奨MVP設計（Vanilla JS ES2020 + CSS Grid + LocalStorage + BroadcastChannel API）をそのまま採用し、追加フレームワークは一切導入しない。

### スコープ外（明示的に除外）

- **クラウド同期・サーバーサイド処理:** オフライン動作必須・CDN不可制約により除外。クラウド連携が必要になった場合は新規Intentとして立てる。
- **レイアウトB（リスト表示）:** MVPはカードグリッド（A）のみ。CEOの承認条件に明示。
- **印刷・レシート出力:** KDSの責務外。POSシステム側の機能。
- **在庫管理・売上集計:** 別システムの責務。KDSはオーダー表示に特化。
- **PWA化（Service Worker）:** 単一HTMLファイル運用で要件を充足するため、MVPでは不要。v2以降で検討。
- **外部API連携（翻訳API等）:** CDN不可・オフライン要件により除外。翻訳テキストは静的に埋め込む。

## UI/UXガイドライン

### レイアウト原則

- カードは横スクロールのグリッドに配置し、画面上部から順に並べる
- カード幅は固定（推奨: 280〜320px）、高さはコンテンツに応じて可変
- 新着オーダーは左端に追加し、古いオーダーほど右に押し出される（時系列順）
- ヘッダーには店名・現在時刻・言語切り替えボタンのみ配置し、情報を最小化する

### カード構造（上から順に）

```
┌─────────────────────────────┐
│ [NEW/予約バッジ]  A1    2名  │  ← 卓番号 28-32px
│ 12:34  ⏱ 07分             │  ← 受付時刻 / 経過時間（色付き）
├─────────────────────────────┤
│ ✓ 土鍋ご飯（白米） × 2     │  ← 完了品目（打ち消し線）
│   辛さ控えめ                │  ← オプション（セカンダリカラー）
│ □ 豚角煮トッピング × 1     │  ← 未完了品目
└─────────────────────────────┘
```

### インタラクション仕様

- **品目タップ:** 即座に打ち消し線を付与・`done: true`を反映。再タップで取り消し可能
- **カード自動消去:** 全品目`done: true`になった時点でフェードアウト（300ms）後に削除
- **言語切り替え:** ヘッダーボタンをタップ→即時全ラベル切り替え→LocalStorageに保存

### アクセシビリティ・視認性

- コントラスト比 4.5:1以上（WCAG AA準拠）
- 厨房の蛍光灯・水蒸気環境を考慮し、グレア低減と高コントラストを優先した産業用ライトテーマ固定（ダークは不採用）
- カード間マージン最低12px（隣接カードの誤タップ防止）
- 経過時間バッジは数値＋背景色の二重通知（色覚サポート）

### 多言語対応

3言語のUIラベル対応表（静的埋め込み）:

| UI要素 | 日本語 | ネパール語（Nepali） | 繁体字中国語 |
| --- | --- | --- | --- |
| 新規 | 新規 | नयाँ | 新訂 |
| 予約 | 予約 | आरक्षण | 預約 |
| 人数 | 名 | जना | 人 |
| 受付 | 受付 | प्राप्त | 接單 |
| 経過 | 経過 | बितेको | 經過 |
| 完了 | 完了 | पूर्ण | 完成 |

- [ ] ヘッダーの言語ボタンをタップすると、全ラベルが日本語→ネパール語→繁体字中国語→日本語の順に切り替わる
- [ ] 言語設定をネパール語にしてブラウザをリロードすると、ネパール語で起動する
- [ ] 2つのブラウザタブでHTMLを開き、片方で品目を完了にすると、もう一方にも3秒以内に反映される
- [ ] HTMLファイルをオフライン（機内モード）で開いても全機能が動作する
- [ ] 卓番号フォントが28px以上、メニュー名フォントが22px以上であることをDevToolsで確認できる
- [ ] 品目タップエリアが64×64px以上であることをDevToolsで確認できる
- [ ] 15件のモックオーダーを同時に`window.KDS_ORDERS`に設定しても、レイアウトが崩れない
