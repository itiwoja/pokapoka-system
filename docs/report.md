## CEO 方針

KDSのUI設計は、厨房オペレーションの効率と顧客体験の品質を直接左右するクリティカルな経営インフラである。「土鍋飯ぽかぽか」のように多言語スタッフ・多国籍客層が混在する小規模店舗では、誤オーダー・提供遅延がそのまま口コミ評価に直結するため、画面の視認性・操作性はコスト以上に重視すべき競争要因だ。本調査では、実際のKDS製品と業界ベストプラクティスを横断的に分析し、Android 2台・オフライン運用という制約内で最大の現場効果を引き出すレイアウト設計の根拠を固める。

## CTO 技術評価レポート

### 結論

HTML/CSS/JS単一ファイル・オフライン制約下では、**レイアウトB（縦リスト）が最小実装コストかつ厨房環境での視認性・信頼性が最高**。ただしCEO判断によりMVPはA（カードグリッド）から実装。

### 推奨技術スタック

| 層 | 選定 | 選定理由 |
|---|---|---|
| フロントエンド | Vanilla JS (ES2020) + CSS Grid | CDN不可制約、ES2020はAndroid 10以降で完全サポート |
| データ層 | LocalStorage (primary) + IndexedDB (bulk) | 5MBで十分（1オーダー≒200bytes） |
| 同期層 | BroadcastChannel API | 同一Wi-Fi内タブレット2台間P2P同期 |
| オフライン | Service Worker + Cache API | 単一HTMLをオフラインキャッシュ |

### 各レイアウト技術評価

| レイアウト | 実装工数 | 推奨度 | 主な懸念 |
|---|---|---|---|
| A: カードグリッド | 12-16時間 | 中（CEOの第一選択） | カード12枚超でスクロール問題 |
| **B: 縦リスト** | **8-10時間** | **最高** | 業界非デファクト、横展開で見劣りリスク |
| C: レーン分割 | 20-28時間 | 低（v1退避） | 非エンジニア運用困難 |
| D: 大文字フォーカス | 14-18時間 | 低 | ピーク時の情報見落としリスク大 |

### Android タブレット最適化仕様

| 要素 | 推奨仕様 |
|---|---|
| テーブル番号フォント | 28-32px |
| メニュー品名フォント | 22-26px |
| タッチターゲット | 最小64x64px（濡れた手・グローブ想定） |
| コントラスト比 | 7:1以上（厨房環境） |
| タップ遅延 | `touch-action: manipulation`で300ms排除 |

### MVP工数見積もり

| フェーズ | 内容 | 工数 |
|---|---|---|
| Phase 1 | HTMLスケルトン + CSS基盤 | 0.25人月 |
| Phase 2 | JSコアロジック（CRUD・LocalStorage） | 0.25人月 |
| Phase 3 | Service Worker + オフラインキャッシュ | 0.125人月 |
| Phase 4 | BroadcastChannel同期 + 2台間動作検証 | 0.125人月 |
| Phase 5-6 | 実機検証 + バグ修正 | 0.25人月 |
| **合計** | | **1.0人月（約40時間）** |

**技術リスク(最重要):** BroadcastChannelは`file://`プロトコルで動作不可→ローカルHTTPサーバー経由が必要（+0.25人月）

### 業界標準の考察

**2025年業界標準:** A型（カードグリッド）が事実上の標準。チケットサイズ（S/M/L）で情報密度を調整するアプローチが主流。
**配色標準:** 背景ダーク、通常=白/薄グレーカード、警告=黄、緊急=赤、完了=グレーアウト
**フォントサイズ標準:** 品目名24px以上、テーブル番号28-36px

## CEO 最終判断

### 最終判断: conditional

### 統合判断

財務的には明確にGO判定。ペイバック2.1ヶ月、3年TCOで229万円の優位、補助金適用後の実質開発費125,000円は健全なキャッシュフロープロファイルを示す。

**CMO（C推奨）とCTO（B推奨）の対立に対するCEO裁定:**

**MVPはAカードグリッドから開始する。**

根拠:
1. 業界デファクトであり、ユーザーが「見覚えのある画面」として受け入れやすい
2. Cレーン分割は「非エンジニア運用困難」が小規模店舗では致命的
3. 将来のSaaS転換時のポジショニング確保

**多言語対応はMVPスコープに含める。**
全競合6社がネパール語未対応という差別化は、MVPから実装するだけで「最初から違う」製品になる。

**Cレーン分割はv1スコープに明示保留。**
PMF検証後にv1で「プレミアムビュー」として実装する。

### 承認条件

| 条件 | 詳細 |
|---|---|
| MVPスコープ固定 | レイアウトA（カードグリッド）+ 3言語（日本語・ネパール語・繁体字中国語）のみ |
| 完了条件測定可能 | 「Android 10インチタブレット上でオーダー追加から画面反映まで1秒以内、3言語切り替えが0タップで自動判定」 |
| 開発契約 | マイルストーン払い条項 + 上限予算395,000円を明記 |

### 優先アクション（3項目）

1. **2週間以内:** MVPスコープを書面確定（レイアウトA + 3言語 + LocalStorage + BroadcastChannel）
2. **MVP完了後30日:** 「土鍋飯ぽかぽか」実稼働データ収集 + 沖縄エリア他店舗2-3軒への無料トライアル提案
3. **週次モニタリング:** オーダー処理遅延件数・スタッフUI不満・多言語切り替え利用率
