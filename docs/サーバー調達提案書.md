# ご提案書：店内システム用サーバーについて

**作成日**：2026年6月18日
**件名**：卓上注文端末・厨房（デシャップ）端末を動かすサーバー機器のご提案

---

## 1. ご提案の概要

店内の卓上注文端末と厨房（デシャップ）端末を動かすための、店内設置サーバーをご提案します。

結論として、**小型の高性能PC1台（GMKtec M6 Ultra、税込 約67,000円）** で必要十分です。大型・高額な専用サーバーは、今回のシステムには過剰であり、コストの無駄になると判断しました。

---

## 2. このサーバーの役割

店内サーバーは、次の3つの仕事を担当します。

1. **予約情報の取り込み** … 予約システム「テーブルチェック」から予約情報を受け取る
2. **注文端末・厨房端末の動作** … 卓上の注文端末と厨房の表示端末を動かす（画面を配信する）
3. **注文情報の受け渡し** … お客様が卓上で注文した内容を厨房の端末へ届ける

お客様の注文は、必ずこのサーバーを経由して厨房へ届きます。サーバーが店内システムの「中継地点」となる構成です。

なお、サーバーから卓上端末・厨房端末への画面配信は、**店内のWiFi（無線LAN）**を通じて行います。そのため、安定した動作には店内のWiFi環境が重要になります（詳細は「6. ネットワーク環境について」をご覧ください）。

## 4. なぜ「大型サーバー」ではなく、この1台で足りるのか

ご検討の過程で、業務用の大型サーバー（10〜30万円台）も候補に挙がりました。しかし、今回のシステムでは以下の理由から不要と判断しています。

- **扱うデータ量がごく小さい** … 予約・注文の情報は1日あたり数MB程度。大きな処理能力や大容量ストレージは必要ありません。
- **サーバー本体にデータを保存しない設計** … 予約情報は「テーブルチェック」側で管理されるため、本体が万一故障してもデータは失われません。
- **店内に常駐させやすい** … 小型・静音で、設置場所や電源を選びません。

つまり、高価なサーバーが持つ機能（大容量・高度な冗長化など）は、今回の用途では「あっても使わない」ものです。その分のコストを抑え、必要十分な構成にまとめました。

---

## 5. 含めなかった機能と、その理由

提案をシンプルにするため、以下はあえて含めていません。判断の根拠を明記します。

| 含めなかったもの | 理由 |
|---|---|
| データ二重化（RAID） | 本体にデータを保存しないため、二重化の必要性が低い |
| バックアップ機器 | 予約データはテーブルチェック側で保管されるため不要 |
| 無停電電源装置（UPS） | 設置環境で停電・電源トラブルの懸念がないとの前提による |

## 6. ネットワーク環境について（重要）

本システムでは、サーバーから卓上端末・厨房端末への画面配信を**店内のWiFi（無線LAN）**で行います。サーバー本体の性能は十分ですが、システム全体が快適に動くかどうかは、**店内のWiFi環境の品質に大きく左右されます。**

WiFiが不安定だと、「注文がなかなか送れない」「厨房の表示が遅れる」といった、営業に直接影響する問題が起こり得ます。安定運用のため、以下の点をご確認・ご検討いただくことを推奨します。

- **業務用のWiFi機器（アクセスポイント）を使用する** … 家庭用ルーター1台では、店内全体に安定した電波が届かない場合があります。
- **お客様用WiFiと、端末用WiFiを分ける** … 同じ回線が混雑すると、端末の動作が遅くなることがあります。
- **店内の広さ・席数に対して、電波が全席に届く配置にする** … 特に厨房は金属什器や電子レンジが電波を妨げやすいため、注意が必要です。

現在の店内WiFi環境によっては、アクセスポイントの増設・更新が別途必要になる可能性があります。サーバー導入とあわせて、一度WiFi環境の確認を行うことをおすすめします。

## 8. 本提案の前提条件

本提案は、以下の前提に基づいています。

- **予約情報**は予約システム「テーブルチェック」が管理します。
- **注文情報**は、厨房（デシャップ）での表示・調理指示にのみ使用し、システムとして保存・蓄積は行いません（使い捨て）。

このため、サーバー本体にデータを保存する必要がなく、データのバックアップも不要です。これが、本提案で小型サーバー1台という構成を可能にしている根拠です。

※将来、売上集計や注文履歴の保存といった機能が必要になった場合は、注文データの保存とバックアップの仕組みを別途追加する必要があります。
