## 1. プロジェクト概要

### 1-1. 背景・目的

沖縄県浦添市のネパール料理店「ぽかぽか」において、以下の課題を解決するためオーダリングシステムを開発する。

| 課題 | 内容 |
|------|------|
| 言語の壁 | スタッフがネパール人・日本人混在、外国人客が多く口頭説明が困難 |
| 提供時間 | 来店から 30 分以内の提供を目標とするが、現状それを超えてしまっている |
| 回転率 | ピーク時に丁寧な説明ができず、スマートな接客が実現できていない |

### 1-2. 開発スコープ（Phase 1）

| 対象 | 内容 |
|------|------|
| ✅ 対象 | 客側タブレット（注文入力）〜 厨房端末（注文受信・調理管理）まで |
| ❌ 対象外 | 会計・POS連携 |
| ❌ 対象外 | 管理端末（メニュー登録・削除・Sold Out 設定・予約管理） |
| ❌ 対象外 | 保守・運用 |

> **Phase 2 以降に持ち越す機能**: 管理端末、予約管理・席ロック、メニュー登録・削除

## 5. 機能要件

### 5-1. 客側タブレット

#### 5-1-1. スクリーンセーバー機能

| # | 機能 | 詳細 |
|---|------|------|
| F-01 | 動画ループ再生 | 無操作時に美味しいメニューの動画（3本程度）をループ再生する |
| F-02 | スクリーンセーバー解除 | 画面タップにより解除し、言語選択画面に遷移する |

#### 5-1-2. 言語選択機能

| # | 機能 | 詳細 |
|---|------|------|
| F-03 | 言語選択 | 日本語 / 英語 / 韓国語 / 中国語（簡体）/ 台湾語（繁体）の 5 言語から選択できる |
| F-04 | 言語保持 | 選択した言語は注文完了まで保持される |

#### 5-1-3. メニュー一覧表示機能

| # | 機能 | 詳細 |
|---|------|------|
| F-05 | カテゴリ別表示 | メニューをカテゴリ別に一覧表示する |
| F-06 | おすすめ表示 | 「本日のおすすめ」をトップに表示する（内容は管理者が日次で更新） |
| F-07 | Sold Out 表示 | 売り切れメニューには「Sold Out」を表示し、注文不可にする |
| F-08 | アレルギー表示 | 各メニューにアレルギー情報アイコン（または記号）を表示する |
| F-09 | 多言語表示 | メニュー名・説明文を選択言語で表示する |

#### 5-1-4. メニュー詳細表示機能

| # | 機能 | 詳細 |
|---|------|------|
| F-10 | 詳細情報表示 | メニュー名・価格・説明・アレルギー情報を表示する |
| F-11 | 動画再生 | 料理の説明動画をインライン再生する |
| F-12 | オプション選択 | オプション（辛さ・サイズ等）を選択できる |

#### 5-1-5. 注文機能

| # | 機能 | 詳細 |
|---|------|------|
| F-13 | カートへの追加 | メニューと数量・オプションをカートに追加できる |
| F-14 | カートの確認・変更 | カートの内容を確認し、数量変更・削除ができる |
| F-15 | 注文確定 | カートの内容を確定し、厨房に送信する |
| F-16 | 注文完了表示 | 注文完了メッセージを表示する |
| F-17 | 追加注文 | 注文完了後も引き続き追加注文ができる |

#### 5-1-6. 卓番号表示

| # | 機能 | 詳細 |
|---|------|------|
| F-18 | 卓番号表示 | 現在の卓番号（例: A1）を常時表示する |

## 6. 非機能要件

### 6-1. パフォーマンス

| 項目 | 目標値 |
|------|--------|
| 注文送信〜厨房端末への反映 | 3 秒以内 |
| 画面初期表示（メニュー一覧） | 3 秒以内 |
| 動画の初回ロード開始 | 5 秒以内 |

### 6-2. 可用性

| 項目 | 内容 |
|------|------|
| 対応環境 | 店内 Wi-Fi 環境下での動作を前提とする |
| オフライン対応 | 対象外（Wi-Fi 環境が前提） |

### 6-3. セキュリティ

| 項目 | 内容 |
|------|------|
| 厨房端末のアクセス制限 | 店内ネットワークからのみアクセス可能にする |
| 客側タブレットの操作制限 | 注文以外の操作（設定変更・他サイトへの遷移等）をできないようにする |

### 6-4. 対応端末・環境

| 項目 | 内容 |
|------|------|
| 客側タブレット | Android タブレット（子機・16台） |
| 厨房端末 | Android タブレット or 既存 PC（2台） |
| ネットワーク | 店内 Wi-Fi |
| サーバー | AWS 上にデプロイ（クラウド構成） |

### 6-5. ユーザビリティ

| 項目 | 内容 |
|------|------|
| 多言語対応 | 日本語 / 英語 / 韓国語 / 中国語（簡体）/ 台湾語（繁体） |
| UI 方針 | 店舗の公式 Web サイトのデザインを基準に統一、ブランディング重視 |
| 客側 UI | 直感的な操作（IT リテラシーを問わず操作できる） |
| 厨房側 UI | 視認性・操作性を特に重視（ピーク時でも即座に確認できる） |

以下は **Phase 1 の開発対象外** とする。

| 機能 | 理由 |
|------|------|
| 会計・POS 連携 | 既存のレジを継続使用するため |

| 予約管理・席ロック | Phase 2 の管理端末機能として実装予定 |

## 9. 業務フロー（Phase 1 対象範囲）

```text
【来店〜提供フロー】

お客様着席
  ↓
スタッフが席に案内し、卓番号を設定したタブレットを手渡す
  ↓
お客様がタブレットで言語を選択
  ↓
お客様がメニューを閲覧・選択し、注文を確定する
  ↓
注文データが厨房端末にリアルタイムで送信される
  ↓
厨房スタッフが注文内容・経過時間を確認しながら調理
  ↓
料理を提供後、厨房スタッフが「完了」ボタンを押す
```

## 11. 用語定義

| 用語 | 説明 |
|------|------|
| 子機 | お客様が注文に使用するタブレット端末（16台） |
| 厨房端末 | キッチンスタッフが注文を確認・管理するための端末（2台） |
| 親機 | 管理者（ホール）が使用する管理端末（Phase 2 対象） |
| Sold Out | 在庫切れにより注文不可となったメニューの状態 |
| 卓番号 | 席を識別するための番号（例: A1、A2 形式） |
| 新規客 | 当日の飛び込み来店のお客様 |
| 予約客 | 事前に予約を入れているお客様 |
| オプション | メニューに付帯する選択肢（辛さ・サイズ等） |
