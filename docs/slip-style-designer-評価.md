## 総合評価: ★★★★☆ (4/5)

「印刷スタイルを直感的に設定できる」という要求に対し、スタンドアロンのプレビュー付き設定ツールとしては十分な完成度。外部依存ゼロ・単一HTML・ES5 スタイルなどプロジェクトの流儀にも沿っている。最大の減点要素は **設定がまだ実際の印刷経路(KDS の伝票プレビュー / relay-server の ESC/POS)に接続されていない** こと。現状は「スタイルを決めるためのデザインツール」であり、「決めたスタイルで印刷されるようにする」には統合作業が別途必要。

## 2. コード品質 — ★★★★☆

**良い点**

- IIFE + `"use strict"`、ES5 スタイル(`var`・`function`)で `kds-a-grid.html` の書き方と一貫している
- DOM 生成は全て `textContent` ベースで XSS の心配がない(`innerHTML` は空文字クリアのみ)
- `localStorage` アクセスを try/catch で包み、プライベートモード等でも壊れない
- `data-key` / `data-group` 属性による宣言的なコントロール束縛で、設定項目の追加が容易
- JSON 読み込み時に `Object.assign({}, DEFAULTS, parsed)` で欠損キーを補完

**問題点・注意点**

- **(実装時に混入・修正済み)** 初版に `@page-size-sync();` という構文エラーになる行が混入していた。動作確認前の Edit で除去済みだが、生成コードは検証必須という教訓
- `render` を末尾で `updatePageRule` 付きにラップし直す書き方はトリッキー。素直に `render()` の中で呼ぶべき
- JSON 読み込みで**値の型・範囲を検証していない**。`paperWidth: 9999` のような値もそのまま適用される(printer.js の `normalizeJob` は丸め込みをしており、姿勢が揃っていない)
- `mmToPx` の係数 3.35 は 96dpi の正確な換算(3.78px/mm)より小さく、プレビュー幅は実寸より約 11% 細い
- `alert()` / `confirm()` を使用。KDS 本体はモーダルダイアログで統一しており不整合(ブラウザ自動化もブロックする)
- サンプル既定値が HTML(`value="12"`)と JS(`loadSample` のフォールバック)で二重定義

## 3. プロジェクト適合性 — ★★★☆☆

- **単一 HTML・外部参照ゼロ・オフライン動作**という KDS 本体の制約を守っている(ファイルを開くだけで動く)
- CSS 変数の色(`#1a1612` 等)や `@media print` の作りは `kds-a-grid.html` の伝票プレビューと揃えてある
- **最大のギャップ: 設定 JSON がどこにも消費されない**
  - `kds-a-grid.html` の伝票プレビュー(`.slip` の CSS)は本ツールの設定を読まない
  - `relay-server/printer.js` の `buildEscPos()` は倍角・太字・罫線・送り行数がハードコードで、スタイル指定を受け取る口がない
  - localStorage キーも `slip_style_designer_v1` と独自で、KDS 側の `kds_*` 系と接続されていない
- 統合するなら: ① KDS が同キーの JSON を読んで `.slip` へ適用、② `POST /api/print` の body に style オブジェクトを追加し `buildEscPos()` が解釈(ESC/POS は等倍/2倍の段階制御なので「S/M/L/XL → 等倍/2倍」のマッピング定義が必要)、の 2 点セットが最小構成

## 4. UX / アクセシビリティ — ★★★☆☆

- 日本語 UI・設定パネルのグルーピング・sticky プレビューは分かりやすい
- レスポンシブ対応あり(760px 以下でプレビューが下に回る)
- **KDS 本体の基準に届いていない点**:
  - セグメントボタンが `min-height: 30px`、削除ボタンが 28px 角 — KDS のタップ下限 44px(#58)未満。厨房のタブレットで使うなら要拡大
  - セグメントボタンに `aria-pressed` がなく、選択状態が支援技術に伝わらない
  - スライダーに `aria-label` や `<output>` の関連付けがない
- ただし本ツールは店舗スタッフが日常操作する画面ではなく、開店前に一度使う管理ツールなので、優先度は KDS 本体より低くてよい

## 5. 動作確認結果(2026-07-17 実施)

Chrome + ローカル HTTP サーバーで確認:

- [x] 初期表示・コンソールエラーなし
- [x] 用紙幅 58mm 切替 → プレビュー幅が即時縮小
- [x] 卓番 XL・店名非表示 → 即時反映
- [x] リロード後も設定保持(localStorage)
- [x] 品目の追加(4件)・先頭削除(3件) → 行とプレビューが正しく同期
- [x] JSON 書き出しクリックがエラーなく実行
- [ ] JSON 読み込み・既定値リセット・テスト印刷は**ネイティブダイアログを開くため自動確認できず**(手動確認を推奨)
- 補足: 確認中に何度かスクリーンショット取得(CDP)がタイムアウトしたが、JS 実行は終始正常応答しており、ページ側のフリーズではなく環境要因

## 改善提案(優先度順)

1. **[高] 印刷経路への接続** — KDS の `.slip` と `buildEscPos()` が設定 JSON を消費するようにする(これがないと「設定できるが反映されない」ツールのまま)
2. **[中] JSON 読み込み時のバリデーション** — `printer.js normalizeJob` と同じ発想で、許容リスト・範囲へ丸める
3. **[中] `render` ラップの解消と `mmToPx` 係数修正**(3.78 に)
4. **[低] `alert`/`confirm` をインライン UI に置換、セグメントボタンへ `aria-pressed` 付与、タップ領域 44px 化**
5. **[低] サンプル既定値の一元化**

## 結論

スタンドアロンの「伝票デザイン検討ツール」としてはこのまま使える品質。ただしプロジェクトの文脈では**設定を実印刷に反映する統合(改善提案1)が本丸**であり、そこまで含めて初めて「印刷スタイルを設定できるアプリ」が完成する。PR を出すなら、本ツール単体を「デザイン検討用」と明記して先行マージし、統合を別 Issue に切るのが現実的。
