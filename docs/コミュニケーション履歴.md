### 2026-05-09（予約システム連携方針の議論）

**壱基** が STUDIO の API 制約を調査し、以下の案を提示：

1. **SaaS 作戦**：外部予約システムを STUDIO に埋め込む（Webhook で API にデータ送信）
   - メリット：品質保証・爆速
   - デメリット：継続課金、SaaS の JSON 構造に依存
2. **自作作戦**：予約サイトを自作して STUDIO に iframe で埋め込む
   - メリット：自分たちで設計・実装できる
   - デメリット：フロントエンドを1から作る必要あり

**小川優斗** が AWS での自作案を提案：

```
STUDIO にある「予約する」ボタン
  ↓
AWS で作った予約専用ページに遷移（または iframe で埋め込み）
  ↓
入力データが AWS 上の DB に保存
```

**壱基** が STUDIO の柔軟性の問題を指摘：
- STUDIO 内で「席が埋まっているため予約不可」を表示できない可能性がある

**隆之介**：最悪手入力でも対応可能。メリデメを次のミーティングまでにまとめる。
