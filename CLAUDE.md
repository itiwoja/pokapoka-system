# CLAUDE.md (pokapoka-system 用プロジェクト指示)

このファイルはリポジトリ直下に置かれ、本リポジトリで作業する AI (Claude Code 等) 向けの指示を定義する。

## Git ワークフロー（必須）

本プロジェクトは AI を使い、**複数セッションを並行させて**開発している。そのため「ちゃんとした作業」は必ず以下の流れで行う。`main` への直接コミット・直接プッシュはしない。

> **最重要: ファイルを編集する前に、必ず作業用のワークツリーを作る。**
> セッションごとに独立したワークツリーを持つことで、別セッションのブランチ切り替えや未コミットの変更に巻き込まれずに作業できる。
> コードやドキュメントの変更が発生する作業を始めるときは、**1 行目の編集より先に**ワークツリー（とブランチ）を作成すること。
> 「小さい修正だから」「あとで切ればいい」は不可。編集を始めてからワークツリーがないことに気づいた場合も、その時点で即座に作ってからコミットする。

1. **ワークツリーとブランチを作る**
   - 作業ごとに `main` から新しいブランチを切り、**専用のワークツリーを作成する**（1 作業 1 ワークツリー）
   - 命名規則: `feature/<内容>` / `fix/<内容>` / `docs/<内容>` / `chore/<内容>`
   - ワークツリーの置き場所はリポジトリの外の `C:\Projects\dev\pokapoka-worktrees\<ブランチ名の / を - にしたもの>`
     ```sh
     git fetch origin
     git worktree add -b feature/xxx ../pokapoka-worktrees/feature-xxx origin/main
     cd ../pokapoka-worktrees/feature-xxx
     ```
   - **作業開始時に必ず実施する。**編集ツールを走らせる前に、今いる場所が今回の作業用ワークツリーかを確認する
     ```sh
     git worktree list           # 既存のワークツリー一覧（他セッションが使用中のものが分かる）
     git branch --show-current   # main なら必ず新しいワークツリーを作る
     ```
   - 既存の作業ワークツリー上にいても、**別テーマの作業を始めるなら新しく作る**（1 ワークツリー 1 テーマ）
   - メインの作業ディレクトリ `C:\Projects\dev\pokapoka-system` は `main` に置いたままにし、そこで直接作業しない
2. **作業・コミット**
   - 作業ブランチ上でコミットを積む（コミットメッセージは `<type>: <説明>` 形式）
3. **PR を出す**
   - `gh pr create` で `main` 向けに Pull Request を作成する
   - PR には変更内容・目的・確認手順（テスト計画）を書く
4. **承認を待つ**
   - レビュー・承認（Approve）が付くまでマージしない
   - CI がある場合はグリーンになるまでマージしない
5. **マージ後、ワークツリーとブランチを削除する（自動で片づける）**
   - マージが完了した時点でそのワークツリーは役目を終える。**ユーザーの指示を待たず、その場で自動的に消す**（残すと `git worktree list` が実態と合わなくなり、同名ブランチの再作成もできなくなる）
     ```sh
     cd C:/Projects/dev/pokapoka-system              # 削除対象のワークツリーから出てから実行する
     git worktree remove ../pokapoka-worktrees/feature-xxx
     git branch -d feature/xxx
     git push origin --delete feature/xxx
     ```
   - `gh pr merge --delete-branch` を使うと merge とリモートブランチ削除を一度に行える（**ワークツリーとローカルブランチの削除は別途必要**）
   - 未コミットの変更が残っていると `git worktree remove` は失敗する。中身を確認し、捨ててよいと確認できた場合のみ `--force` を付ける
   - ディレクトリを手で消してしまった等で一覧がズレた場合は `git worktree prune` で整理する

### マージ済みワークツリーの自動片づけ

自分がマージしたぶんだけでなく、**作業開始時に `git worktree list` を見た際、マージ済みブランチのワークツリーが残っていたら自動で片づける**（過去のセッションが消し忘れたものが溜まるため）。

- 判定は「そのブランチが `origin/main` にマージ済みか」で行う
  ```sh
  git fetch origin
  git branch --merged origin/main    # ここに出るブランチはマージ済み = 片づけ対象
  ```
- ただし以下は**触らない**。他セッションが作業中の可能性があるため
  - 未コミット・未プッシュの変更が残っているワークツリー（`git -C <path> status --porcelain` が空でないもの）
  - マージされていないブランチ
- 片づけは `git worktree remove`（`--force` なし）+ `git branch -d`（`-D` なし）で行う。この 2 つは安全側に倒れており、変更が残っていれば失敗して止まるので、**マージ済みと確認できていれば確認不要で実行してよい**
- 失敗したものは無理に消さず、ユーザーに報告する

### AI が守ること

- **コード・ドキュメントの編集を伴う作業に入る前に、必ずワークツリーとブランチを作る。**ユーザーから「ワークツリーを作って」と言われなくても自律的に行う（作成は非破壊的な操作なので確認不要）
- 他セッションが並行して動いている前提で、作業開始前に `git worktree list` を確認し、**他のワークツリーやそのブランチには触らない**（例外はマージ済みワークツリーの自動片づけ。既に役目を終えているため）
- 明示的な指示がない限り、上記フローを勝手に省略しない（`main` に直接 push しない）
- 破壊的操作（`push --force`, `branch -D`, `reset --hard`, `git worktree remove --force` など）は必ずユーザーに確認してから実行する
- PR 作成後、承認が下りていない段階で自分からマージしない
- **マージ済みブランチのワークツリー・ローカルブランチの削除は、ユーザーの指示がなくても自律的に行う**（破壊的操作ではあるが、マージ済みかつ変更が残っていなければ安全なため）。マージ直後の自分のぶんと、作業開始時に見つけた残骸の両方が対象

## スコープ

本リポジトリのスコープは「厨房端末（KDS）画面」のみ。客席タブレット・管理画面・予約連携などは対象外（詳細は README.md 参照）。
