## 2. プロジェクト概要

### 2.1 目的

* **スマートな店舗運営の実現**：言語の壁を解消し、少人数での効率的なオペレーションを確立する。
* **回転率の向上と提供遅延の解消**：ピーク時の提供時間（来店から30分以内）を遵守し、顧客を待たせない体制を作る。
* **価格に見合った接客の維持**：接客の質を落とさず、動画等で食べ方を分かりやすく案内する。

### 2.2 背景

* 店舗規模の拡大に伴い、言語の壁やピーク時のオペレーション負荷が増大している。
* 現在はスタッフ（ネパール人2名、日本人）に対し、客層の半分が外国人（うち8割が台湾人、その他アジア・欧米圏）であり、一人ひとりへ丁寧な説明をする余裕が不足している。

### 2.3 スコープ（2026-05-21 確定）

開発対象は以下の **2システムに限定** する。

| 区分           | システム                       | 備考                                       |
| :------------- | :----------------------------- | :----------------------------------------- |
| ✅ 対象（Phase 1） | 顧客用タブレットシステム（客側）       | メニュー表示〜カート〜注文確定             |
| ✅ 対象（Phase 1） | デシャップシステム（オーダー表示・配膳指示） | 注文受信〜カード表示〜配膳順並替〜完了操作 |
| ❌ 対象外         | 予約管理                       | **外部SaaSを別途利用**。自社システムとの連携は行わない |
| ❌ 対象外         | 会計・POS連携                  | 既存レジを継続使用                         |
| ❌ 対象外         | 親機（管理端末）               | Phase 2 以降に持ち越し                     |
| ❌ 対象外         | 学校としての保守・運用         | 運用保守企業を別途選定                     |

> **Phase 2 以降に持ち越す機能**: 親機（管理端末、メニュー登録・削除、Sold Out 切替、本日のおすすめ編集、席ロック等）。

## 4. 現状（As-Is）

* **現在の業務フロー：** 予約はシステム等から紙に出力し、専用ボードに貼付して管理。新規客も含め、スタッフが対面でメニュー説明・注文受付を行う。
* **使用しているツール・システム：** 電話での予約システム、紙の予約ボード、既存の会計システム、紙のシフト管理。
* **課題点：** 外国人客（全体の約5割）への説明に時間がかかる。注文受付の負荷が高く、注文から提供まで30分以上待たせてしまうことがある。
* **イレギュラー対応：** 言語の壁による聞き間違いや、ピーク時のオーダー混雑による提供遅延。
* **店舗規模：** 収容人数30名（最大15組）、1人用：8席 / 2人用：3卓 / 4人用：3卓。ピーク時2〜3回転。
* **スタッフ構成：** ネパール人 2名 + 日本人

## 5. 理想像（To-Be）

### 5.1 目指す業務フロー

* **予約：** 外部SaaSで受付（本システムとは独立）。
* **予約客：** 来店後、スタッフが席に案内し、客側タブレットで注文。
* **新規客：** 来店時、スタッフが席に案内し、客側タブレットで注文。
* **注文：** 多言語対応タブレットでメニューを閲覧、おいしい食べ方を動画で説明、アレルギー情報を確認しながら注文。
* **デシャップ：** デシャップ端末でリアルタイムに注文を受信し、新規 / 予約を色分け表示で判別、経過時間とコンロ状態を見ながら調理・配膳指示を行う。

### 5.2 解決したい課題

* 注文受付の無人化による言語の壁の解消、および調理着手の迅速化。
* デシャップオペレーションの可視化（経過時間・優先順位・コンロ管理）によるピーク時提供遅延の削減。

### 5.3 成果指標（KPI）

* 来店から提供まで **30分以内** を遵守する割合（現状の数値計測は今後検討）。
* その他指標は今後設定。

#### 7.3.2 予約フォーム要件（外部SaaS側、もしくは自作予約フォーム）

オーナー要望（2026-05-17、docs/memo.md）に基づき、以下の入力項目と注意事項を持つ予約フォームを用意する。**本システムの開発スコープ外**だが、運用上の前提として記録する。

##### 入力項目

1. 日時
2. 人数

* 大人の方 **1人1膳のご注文必須**（子供とのシェアは可）
* 提供まで **25〜30分** かかること
* 席の状況次第で、予約時間から **1組1時間半** のご利用になること

### 8.2 マスタ一覧

* **座席マスタ：** カウンター8席、2名席3卓、4名席3卓（卓の呼び方は「A1」形式等、システム側で別形式も可）
* **メニューマスタ / オプションマスタ**

## 9. 非機能要件

### 9.1 性能要件

| 項目                             | 目標値    |
| :------------------------------- | :-------- |
| 注文送信〜デシャップ端末への反映 | 3 秒以内  |
| 画面初期表示（メニュー一覧）     | 3 秒以内  |
| 動画の初回ロード開始             | 5 秒以内  |
| 全体レスポンス方針               | ストレスのない迅速なレスポンス（速いほど望ましい） |

### 9.2 可用性

* 営業時間中（特にピーク時）にシステムがダウンしない安定性を確保する。
* 対応環境：**店内 Wi-Fi 環境下** での動作を前提とする。
* オフライン対応：**対象外**（Wi-Fi 環境が前提）。

### 9.3 セキュリティ要件

* **デシャップ端末のアクセス制限：** 店内ネットワークからのみアクセス可能にする。
* **客側タブレットの操作制限：** 注文以外の操作（設定変更・他サイトへの遷移等）をできないようにする（キオスクモード等）。
* **親機（Phase 2）：** スタッフ・管理者のみがアクセスできるよう認証・アクセス制限を行う。

### 9.4 対応端末・環境

| 項目             | 内容                                                                   |
| :--------------- | :--------------------------------------------------------------------- |
| 客側タブレット   | **Android タブレット**（子機・16台）                                   |
| デシャップ端末   | **Android タブレット** or 既存 PC（2台）                               |
| 親機（Phase 2） | 1台                                                                    |
| ネットワーク     | 店内 Wi-Fi（新規 Wi-Fi ルーターを調達）                                |
| サーバー         | **AWS**（クラウド構成）。**ローカルサーバー** も併用検討（2026-05-21） |

### 9.5 運用・保守要件

* 学校としての保守運用は対象外とする。
* 運用保守企業の選定は、田島先生と相談のうえ決定する。

### 9.6 ユーザビリティ

| 項目             | 内容                                                                       |
| :--------------- | :------------------------------------------------------------------------- |
| 多言語対応       | 日本語 / 英語 / 韓国語 / 中国語（簡体） / 中国語（繁体）                   |
| UI 方針          | 店舗の公式 Web サイトのデザインを基準に統一、ブランディング重視            |
| 客側 UI          | 直感的な操作（IT リテラシーを問わず操作できる）                            |
| デシャップ UI    | 視認性・操作性を **特に重視**（ピーク時でも即座に確認できる、特にこだわる）|

### 9.7 拡張性

* 将来的な機能追加（Phase 2 以降の親機・将来拡張項目）を想定した設計とする。
