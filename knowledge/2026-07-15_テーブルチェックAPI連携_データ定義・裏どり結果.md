# テーブルチェックAPI連携：データ定義・裏どり結果（2026-07-15）

> 目的：TableCheck API から KDS にメニュー予約を表示するために「何のデータを・どのAPIで・どう受けるか」を1枚に集約する。
> 位置づけ：[2026-07-15 取得したい情報洗い出し](2026-07-15_テーブルチェックAPI連携_取得したい情報洗い出し.md) の続編。TableCheck 公式ドキュメント（Confluence）での裏どり結果と、KDS 実装（`kds-a-grid.html`）の実コード精査を反映した改訂版。
> 対応 Issue：#74（料金・データ形式・通知メール可否）

---

## 0. 前提：ローカルKDSとクラウドのTableCheckはどう繋がるか（1分で理解する用）

```
【クラウド】                【店内（ローカル）】
TableCheck ◀──30秒ごとに──  中継サーバー（ミニPC） ──LAN/Wi-Fi──▶ KDS（タブレット/パネル）
（予約DB）    取りに行く      reservations 等を保持        window.KDS_ORDERS /
              (pull)         当日分のみ                    kds_stock_v1 を1秒ポーリング
```

- **KDSタブレットが直接TableCheckを叩くことはない**。KDSはオフライン前提の単一HTMLで、店内の中継サーバー（ミニPC）だけを見る。
- **TableCheck側から店内に通知を送り込むこと（Webhook = push）はできない**。店内サーバーはNAT/ファイアウォールの内側にあり、外から到達できないため（2026-06-04 検討済み・裏どりでも確定）。
- よって通信は**店内 → TableCheck の外向き（pull / ポーリング）一択**。TableCheck 側にはこの方式が正式に用意されている（§3）。
- 中継サーバーの役割：①TableCheckから予約を取ってくる ②DB（reservations / reservation_items）に貯める ③KDSが読める形に変換して配る。この設計書は未作成（→ 別途 `docs/relay-server-design.md` を起こす）。

## 1. 使うAPIコンポーネント（⚠️ここを間違えない）

| コンポーネント | 用途 | 備考 |
| --- | --- | --- |
| **Booking v1** (`/api/booking/v1/`) | 予約データの取得。`GET /reservations?start_at_min=...&start_at_max=...`（ページネーション付き） | 1stパーティ（店舗運営者自身とその開発パートナー）のみアクセス可 = 当店は該当 |
| **Sync v1** (`/api/sync/v1/`) | 予約の作成・変更・削除の検知。`GET /sync_events?deliver=true` をポーリング | Webhookも存在するがローカル構成では使えない（§0） |
| （参考）Messaging v1 | 客へのテキストメッセージ配信 | Issue #74 の「予約通知メール可否」はこのAPIの契約可否として確認する |

- ❌ **Site Controller v1（`/api/site/v1/`）は対象外**。予約サイトアグリゲーター向けの店舗設定・卓情報の読み取り専用APIで、予約・メニュー取得の用途ではない（一度リンクを共有したが用途違いと判明）。
- APIコンソール（OpenAPI仕様）: https://api.tablecheck.com/api/booking/v1/docs / https://api.tablecheck.com/api/sync/v1/docs

## 2. 取得したいデータ項目と裏どり結果

### 2-1. Must（これが無いと連携する意味が薄い）

| # | 項目 | TableCheck側 | 裏どり結果 | 当店側の格納先 | KDS側の受け皿 |
| --- | --- | --- | --- | --- | --- |
| 1 | 予約ID | SyncEvent の `syncable_id` / 予約オブジェクトの `id` | ✅ 取得可能（確定） | `reservations.external_id`（冪等キー） | `stock[].rid` |
| 2 | 来店予定日時 | `start_at`（ISO8601・タイムゾーン付き。例 `2020-02-12T19:30:00+0900`） | ✅ 取得可能（確定）。JSTへの変換処理が必要 | `reservations.expected_at` | `stock[].time` |
| 3 | 人数 | `pax`（合計） | ⚠️ **合計は確定**。大人/子供の**内訳フィールドは公開資料で未確認** → 打合せ確認 | `reservations.adults` / `kids` | `stock[].adults` / `kids` |
| 4 | 予約者名 | スキーマ未公開（APIコンソール内） | ⚠️ 取れる可能性は高いが要確認 | `reservations.customer_name` | `stock[].name` |
| 5 | 予約ステータス | `seated` ステータス・キャンセルAPI（`cancel_reason_type` / `cancel_reason`）・SyncEvent の `event_type`（created / updated / deleted）を確認済み | ✅ 変更・キャンセル検知は成立（enum全値は要確認） | `reservations.status` | ストックからの除去で反映 |
| 6 | **事前メニュー明細**（品名・数量） | 🔴 Booking API の制限事項に「メニューアイテム（dining experience）の予約は非対応。回避策としてメモ欄（memo）に自由記述」と明記 | 🔴 **最重要リスク**。※この記述は「API経由での予約**作成**」の制限。TableCheck自身の予約ページで客が選んだコースが `GET /reservations` で**構造化されて返るか**は未確認 → 打合せ/コンソールで最優先確認 | `reservation_items.item_name` / `qty` | `stock[].menu[].name` / `qty` |

**#6 の分岐**：
- 構造化データで取れる → そのまま `reservation_items` にマッピング（理想）
- memo の自由テキストのみ → (a) 中継サーバーでパース（品名・数量の切り出し）＋メニューマスタ（`menu_items.name` / `item_code`）との突合、または (b) 予約メモの記載フォーマットを店側で統一する運用ルール
- 全く取れない → 「メニュー予約」区分が不成立。日時・人数・名前のみの部分連携＋メニューは手動入力のハイブリッド（既存メモ §4 のフォールバック）

## 3. 確定した技術仕様（裏どり済み）

| 項目 | 確定内容 |
| --- | --- |
| 取得方式 | **ポーリング（pull）で成立**。`GET /sync_events?deliver=true` → 返ってきた `syncable_id` で `GET /reservations/{id}`（2段構え。SyncEvent自体に予約データは入っていない） |
| ポーリング間隔 | **30秒に1回より高頻度は不可**（TableCheck指定）。時間範囲指定は最大24時間・最大1000件/クエリ・ページネーション無し |
| イベント処理 | `created` / `updated` は **upsert**（IDが無ければ作成・あれば更新）で統一処理を推奨。`deleted` は該当ID無しなら無視。404が返ったら削除済みとみなす |
| 認証 | 環境ごとに発行される **secret_key**。全リクエストHTTPS必須。JSON形式のREST API |
| レート制限 | コンポーネントごとに **10,000リクエスト/5分**。超過で HTTP 429。当店規模では問題なし |
| テスト環境 | 基本は本番のみだが、**依頼すればTest/Liveの別アカウントを用意してもらえる**（api@tablecheck.com） |
| URL | **https://api.tablecheck.com/api/...** を使う。旧 tablesolution.com 系は2026年廃止（2026-06-03以降は308リダイレクト） |
| 料金 | 公開料金表なし。「料金は変動し、申請プロセス内で個別提示」→ Issue #74 のとおり打合せでしか確定しない |
| アクセス申請 | api@tablecheck.com へメール → APIポータルで利用目的・構成図・月間リクエスト数見積を提出 → 承認後にクレデンシャル発行。**Shopごとにアクセス申請が必要** |

## 4. KDS側の受け皿（中継サーバーが出力すべき形式）

### 4-1. 予約ストック `kds_stock_v1` ← TableCheck連携の直接の流し込み先

```js
[{
  rid: "予約ID",                    // TableCheck予約IDをそのまま or マッピング
  time: "18:30",                    // 来店予定（start_at をJST変換）
  adults: 2, kids: 1,               // 内訳（取れなければ合計をadultsに）
  name: "山田",                     // 予約者名
  menu: [{                          // 事前メニュー。空配列の予約はストックに載せない
    name: "山城牛の焼きすき土鍋御膳",
    qty: 2,
    options: "ご飯大盛り" | null,   // ※受け口は実装済み
    allergies: "えび" | null        // ※受け口は実装済み
  }],
  seenAt: 1234567890                // 取込時刻（KDS側で付与）
}]
```

- **status / updated_at はKDSに持たせない**。キャンセル・変更は中継サーバー側で判断し、ストックからの削除・上書きとして反映する。

### 4-2. 注文カード `window.KDS_ORDERS` ← 予約は「着手」操作でこちらに変換される

```js
[{
  id: "res-" + rid,                 // 予約由来は "res-" プレフィックスで紐付け
  table: "5" | null,                // 来店前はnull（「卓: —」表示）
  type: "reserved",                 // 予約由来は "reserved"（カード金色）
  start: Date.now(),                // 着手時刻
  people: 3, adults: 2, kids: 1,    // 「N名 幼M」表示のため内訳も保持
  items: [{ name, qty, options, allergies, done: false }]
}]
```

変換はKDS内の `moveResToMain()` が担うため、**中継サーバーは 4-1 の形だけ作ればよい**。

## 5. 打合せ / APIコンソールで潰す残課題（優先順）

1. 🔴 `GET /reservations` のレスポンスに**コース/メニュー情報が構造化されて入るか**（§2-1 #6 の分岐がすべてここで決まる）
2. ⚠️ **大人/子供の内訳**フィールドの有無（無ければ「大人1名1膳」バリデーションの設計変更）
3. ⚠️ アレルギー・要望が**どのフィールドで返るか**（専用フィールドか memo 混在か）
4. 料金（初期/月額/従量・プラン差）
5. 予約通知メールの可否（Messaging v1 の契約要否として確認）

→ 1〜3 は https://api.tablecheck.com/api/booking/v1/docs をブラウザで開き、`listReservations` のレスポーススキーマを見れば打合せ前に自己解決できる可能性あり。
