# 予約席・座席DBの設計判断（2026-06-03 打合せ整理）

> 「予約席をどう管理するか」を詰めた際の設計判断と根拠をまとめる。
> 端末フリーアドレス問題 → 予約時点で席確定 → 席ロック（可動）→ 炊飯先行着手のための予約カード移動、までを一連の決定として記録する。
> 一部は記録に無い新仕様（こちらからの提案）。⚠️ は次回打合せ（2026-06-04 現場）でオーナー確認が必要な事項。

---

## A. 客席タブレットはフリーアドレス → 卓番号はスタッフが手渡し時に手動入力（確定）

- 客席タブレットは**席固定ではなくフリーアドレス**運用。
- そのため **端末ID ↔ 席 の紐付けは使えない**（どの端末がどの席にあるか機械的に分からない）。
- 端末は「今どの席か」を自力で知る手段がなく、客の操作からも「誰が／どの席か」は送信されない。
- → **席を教えられるのは端末を渡すスタッフだけ**。スタッフが手渡し時に**卓番号を端末へ入力してから渡す**運用が必須。

**Why:** フリーアドレスである以上、席情報は人間（スタッフ）が手渡し時に与えるしかない。端末固定・端末ID方式は前提が崩れて成立しない。
**How to apply:** 座席DBに「端末ID ↔ 席」の対応は**持たない**。端末は「渡されるたびに卓番号を教えられる入力機」として扱う。手渡し時の卓番号入力UIを客側タブレットに用意する（[打合せ決定事項.md](../docs/打合せ決定事項.md) L148「座席番号を入力後にタブレットを手渡し」と整合）。

## B. 予約時点で「枠」を確保、卓番号は来店時に確定（遅延割当）（2026-06-04 更新）

> ⚠️ **重要な設計修正（2026-06-04）**：当初 B は「予約時点で卓番号まで確定」としていたが、運用実態と矛盾するため**遅延割当モデルに改める**。
> 矛盾の正体：オーナー要望は「**予約客は席が取れることは確定**」だが「**他の客との兼ね合いで、どの卓に座らせるかは事前に決めたくない（柔軟に席対応したい）**」。＝予約時に物理卓を1点固定すると柔軟性が死ぬ。

**`予約枠の確保` と `物理卓番号の割当` を2段階に分離する**（これが「タイミングがむずい」問題の解）：

| 概念 | いつ確定 | 変更可否 | 役割 |
| --- | --- | --- | --- |
| **予約枠の確保**（例：4名・12:00・4名席を1卓ぶん） | **予約時**（真理子さん） | 基本固定 | 席は必ず取れる＝オーバーブッキング防止 |
| **物理卓番号の割当**（例：12卓） | **来店時**（ホール） | 来店まで未割当・柔軟 | 実際どの席に座るか。他客との兼ね合いで当日決める |

- 予約時に決めるのは「**席タイプ×個数の枠を押さえる**」まで。**物理卓番号は未割当（空欄）**で予約成立。
- 事前メニュー予約は来店前から**デシャップ予約ストックにカードが出る**が、その時点では**卓番号は空欄（"卓: —"）**で表示。
- **来店時にホールが卓番号を入力した瞬間、予約ストックカードの卓番号も同じ番号で埋まり一致する**（＝オーナー要望「座った席をデシャップ予約ストックの卓番号と一致させる」を満たす）。

### 来店時の予約特定方法：A案（確定）
卓番号を空欄にした分、来店時に「**どの予約客が来たか**」をホールが特定する操作が要る。
- **A案**：来店した予約客を**ホールが予約一覧から選ぶ → 卓番号を入力 → その予約に紐付く**。
- （B案＝予約ストックの未着席カードをタップして卓入力、も可。実装時にUIを詰める）

### ⭐ A案の操作主体＝ホール座席GUI（ホール端末）に実装（確定）
A案の①〜③は**すべてホール座席GUI（ホール側の画面・端末）で完結**する。厨房のデシャップ画面・客席タブレットには持たせない。

| 操作 | 画面 |
| --- | --- |
| ① 未着席の予約一覧から来店客を選ぶ | ホール座席GUI |
| ② 席マップ（空き/予約ロック/着席）を見て座らせる卓を決める | ホール座席GUI |
| ③ 卓番号を入力 → 座席DB更新＋デシャップ予約ストックの卓番号が埋まる | ホール座席GUI |

- 整合：[議事録 §1](minutes_2026-06-04.md) の権限分け「**厨房＝タップのみ／ホール＝入力機能あり**」と一致。卓割当という「入力」はホール権限の機能。
- 席ロックの付け替え（C項）・席移動も同じホール座席GUIの操作系で扱う（操作の所在を1画面に集約）。

**Why:** 「予約客は席が取れる（確定）」と「どの卓かは当日柔軟に決めたい」は、卓番号を**来店時まで遅延確定**すれば両立する。予約時に物理卓を1点固定する必要は無く、固定すると柔軟な席対応（C項）と衝突する。
**How to apply:**
- 座席DBの `紐付け予約ID` と予約側の `割当卓番号` は **来店時に書き込む**（予約時は null）。予約枠は別途「予約データ（席タイプ・人数・時刻）」で確保。
- デシャップ予約ストックカードは卓番号 null を許容し、来店時の入力で反映する。
- 来店時のホール操作＝「予約を選ぶ → 卓番号入力」を1セットにする（A案）。これで卓番号未割当でも予約⇔卓の一致が保てる。
- ⚠️ 旧記述「予約時点で卓番号確定」「○○様＝○○席が最初から決まっている」は本更新で**無効**。[業務フロー.drawio](../docs/業務フロー.drawio) も「人数に合う席を確定」→「席タイプの枠を確保（卓番号は来店時）」に要修正。

## C. 席ロックは確定で実装。ただし「固定」ではなく「可動（再配置可能）」（確定）

- B で予約席を確保する際、その席を**ロック**して新規客の割当を防ぐ（[打合せ決定事項.md](../docs/打合せ決定事項.md) L138, L141「4名が12:00予約→4人席を11:10〜ロック」）。
- **席ロックは確定で実装する**（オーナー方針）。
- ⚠️ ただしクライアントから「**柔軟に席の配置を変えたい**」要望あり。一度ロックした予約席でも、運用判断で**別の席へ付け替え**でき、元席を新規客に開放できる必要がある。
- → 紐付けは**固定（焼き付け）ではなく更新可能（mutable）**。「予約席の移動」操作で `席.紐付け予約ID` を別席へ移し替える（移動元は空き＝新規割当可に戻る）。

**Why:** 予約席固定だけだと現場の柔軟な席運用に合わない。予約客を別席に動かして元席を回せるようにすることで、回転率と現場判断の自由度を確保する。
**How to apply:** 座席DBの `紐付け予約ID` は更新可能前提。「予約席の移動」は ①移動元を空きに戻す ②移動先をロックする、を1操作（1トランザクション）で行う。デシャップ画面のカード移動操作（D項）と同じ操作系で扱う。

## C-2. 予約を「メニュー有無」で2分岐して扱う（確定）

予約を `事前メニューの有無`（[要件定義（仮）.md](../要件定義（仮）.md) L273）で分け、デシャップでの扱いを変える。

| 予約タイプ | デシャップ予約ストック | 理由 | 注文の発生 |
| --- | --- | --- | --- |
| **事前メニュー予約あり** | ✅ 載せる | 何を炊くか分かる → 来店前に先行着手（炊飯）できる（E項） | ストック表示→30分前プロンプト→着手で進行中へ |
| **席だけの予約（メニュー未定）** | ❌ 載せない | 注文は**着席後にタブレットで行う** → 来店前にデシャップでやることが無い | 着席後に通常の新規注文として発生し、その時に初めてデシャップへ表示 |

- 「席だけの予約」は**席ロックだけしておき、予約自体はデシャップに出さない**。注文は着席後に通常フローで発生させる。

**Why:** メニュー未定の予約をデシャップに出しても厨房は何も着手できず、カードがノイズになる。注文は着席後に必ずタブレットで発生するので、その時点で通常オーダーとして表示すれば足りる。
**How to apply:** デシャップ予約ストックは「items（メニュー）が空でない予約」のみ表示する。**現状モックがこの挙動で実装済み**（[DessupDashboard.tsx](../mockup/飲食店オーダー管理画面/src/app/components/DessupDashboard.tsx) L492 `reservations.filter((r) => r.items.length > 0)`）。席だけの予約のロックは下記 C-3 の座席管理GUIで可視化・管理する（デシャップには依存しない）。

> 📌 **2026-06-04 再確認**：オーナー発言「席だけの予約は予約ストックにのせる必要が無い」で本判断が追認された（設計変更なし）。新規客／席予約／メニュー予約の3分類での整理は [meeting_2026-06-04_questions_for_owner.md](meeting_2026-06-04_questions_for_owner.md) §D 参照。

## C-3. 座席管理GUI を「ホール側機能」として新規スコープ追加（確定・スコープ変更）

- C-2 で「席だけの予約」をデシャップに出さない → **席ロックがデシャップ上で可視化されない** → 新規客との衝突を防ぐため、**席の空き/予約ロック/着席を一覧で見て操作する画面が別途必要**。
- 当初、座席管理システムのGUIは**スコープ外**だったが、本方式を採る以上は**必然的にやらざるを得ない**。→ **ホール側の機能として実装する**（オーナー方針）。
- これにより Phase 1 構成が **2系統 → 3系統** に拡大：
  1. 客席タブレット（注文）＝お客様
  2. デシャップ（オーダー表示）＝厨房
  3. **座席管理GUI（席ロック・空き状況・予約席の付け替え）＝ホール** 🆕

**Why:** 席だけの予約をデシャップから外す方針の必然的帰結として、席ロック状態を見る/操作する場所が要る。それは厨房ではなくホールの業務。座席DB（A・B・C）の操作主体＝この画面。
**How to apply:**

- 座席管理GUIは A（卓番号運用）・B（予約席確定）・C（席ロック＝可動）の操作画面とし、**座席DBの操作主体**とする。
- 機能：席一覧（空き/予約ロック/着席の状態表示）、予約席のロック/解除、予約席の別席への付け替え（C項）。
- ⚠️ スコープ拡大に伴う工数増・端末（ホール用の操作端末/モニター）を見積り直す。[meeting_2026-05-21_scope_hardware.md](meeting_2026-05-21_scope_hardware.md) の「2システム限定」スコープを更新する必要あり。次回打合せでオーナー・先生に共有・確認。
- ⚠️ **2026-06-04 にスコープ表現の衝突が判明**：オーナーは「予約システム／席管理システムは（台帳本体として）スコープ外」と発言。本 C-3 の「座席管理GUI＝実装範囲」と言葉の上で衝突する。「台帳本体＝外部/手運用、デシャップ用の最小座席DB＋ホールGUI＝実装」の境界を明示的にすり合わせる → [meeting_2026-06-04_questions_for_owner.md](meeting_2026-06-04_questions_for_owner.md) §0。

## E. 予約客の「炊飯先行着手」のための予約カード移動（新仕様・提案）

> 記録に無い新仕様。土鍋飯の商材特性（炊き上げに時間がかかる）と「来店から30分以内提供」の課題に整合する提案。

- 目的：**予約客の来店時に、即・炊き上がった米を出したい**。
- 土鍋飯は炊飯に時間がかかる（提供まで 25〜30分：[要望.md](../docs/要望.md) L25）→ 来店時刻から逆算し、**来店前に炊き始める**必要がある。
- 炊き始める＝厨房着手＝その予約は「進行中」になる → デシャップの**予約ストック → 進行中（メイン）へ移動**する必要がある。

### 移動トリガー：方式C（ハイブリッド）で確定
- **来店予定時刻の約30分前**に、システムが「**そろそろ炊くべき予約**」として**通知／プロンプトを出す**（自動では進行中に乗せない）。
- スタッフが**「着手」ボタンを押した時**に、予約カードを進行中（メイン）へ移動する。
- （A=時刻で全自動、B=完全手動、の中間。フードロスと炊飯キャパを吸収するため C を採用）

**Why:** 全自動（30分前に無条件で進行中＆炊飯着手）はノーショー・遅刻でフードロスを生み、土鍋の同時炊飯キャパも超えうる。逆に完全手動だと炊き始めの判断が属人化・失念リスク。30分前に促し＋着手はスタッフ判断、が現実解。
**How to apply:**
- 予約データ／座席DBに **来店予定時刻** と **炊飯リードタイム（既定30分）** を保持し、`来店予定 − リードタイム` で「炊き始め推奨時刻」を算出。
- その時刻を過ぎた予約ストックカードに**「そろそろ炊く」プロンプト**を表示。
- 「着手」ボタン押下でストック→進行中へ移動（D項のボタン移動操作と統一）。

### ⚠️ 次回確認・設計で詰める論点
1. **事前メニュー予約に限定**：何を炊くか分かっている予約（[要件定義（仮）.md](../要件定義（仮）.md) L273 `事前メニューの有無`）でないと先行着手できない。メニュー未定の予約は対象外。
2. **ノーショー／遅刻時のフードロス**の扱い（先行着手後に来なかった場合）。
3. **炊飯キャパ超過**：同時刻帯に予約が集中した場合、土鍋の数で同時着手数に上限。プロンプトの出し方を要検討。
4. **リードタイム 30分** はメニュー・状況で可変か固定か。

> 📌 **2026-06-04 再確認**：オーナー発言「予約客来店時に即炊き上がる米を出したい、炊き始めたタイミングで予約ストック→進行中へ移動」で、本項の方式C（30分前プロンプト＋着手ボタンで移動）が裏付けられた。
>
> 📌 **2026-06-04 確定（やること決定）**：この先行着手は **元々の店の運用**。予約客は**来店の約30分前から米を炊き始める**ことで、炊飯にかかる約30分を吸収・効率化していた（＝新仕様の発明ではなく、現状運用のデジタル化）。よって **30分前通知→着手で進行中へ移動（方式C）を実装することは確定**。
> - 上記残課題1（事前メニュー予約限定）は前提として維持。
> - 残課題2〜4（ノーショー時フードロス・炊飯キャパ超過・リードタイム可変）は**実装を止める要因ではない**。現状運用を踏襲しつつ、運用ガードは実装の中で詰める（着手はスタッフ判断＝方式Cがそのまま吸収する）。

## 座席DB ドラフト（叩き台）

A〜E を踏まえたテーブル案。端末IDは持たない。**B（遅延割当）に合わせ、座席テーブルとは別に予約枠を持つ**。

### 座席テーブル（物理席の現在状態）

| 項目名 | 型 | 必須 | 説明 |
|--------|----|----|------|
| 座席ID | Varchar | ○ | 席を一意に識別（例: `A4`）。卓番号と同一 |
| 席タイプ | Varchar | ○ | `counter`(1名) / `table2`(2名) / `table4`(4名) |
| 定員 | Integer | ○ | 1 / 2 / 4 |
| 表示名 | Varchar | ○ | 画面表示用ラベル |
| 状態 | Varchar | ○ | `empty`(空き) / `reserved`(予約ロック中) / `occupied`(着席中) |
| 紐付け予約ID | Varchar | － | この席に割当てた予約。null＝空き。**来店時に書き込む**（予約時はnull）。**更新可能**（C: 席の付け替え） |
| ロック開始時刻 | Datetime | － | 物理席を予約ロックした開始時刻。**来店時/直前に席を充てた時に入る**（任意） |

### 予約枠テーブル（卓番号に縛られない予約の確保）🆕

> B の「予約時は枠だけ確保、卓番号は来店時に確定」を表現する。物理卓を決めずに「席が取れること」を保証する側。

| 項目名 | 型 | 必須 | 説明 |
|--------|----|----|------|
| 予約ID | Varchar | ○ | 予約を一意に識別 |
| 予約者名 | Varchar | ○ | 来店時の特定に使う（A案：一覧から選ぶ） |
| 来店予定時刻 | Datetime | ○ | 炊飯先行着手（E）の起点。`来店予定 − リードタイム`で炊き始め推奨 |
| 人数（大人/子供） | Integer | ○ | 「大人1名1膳必須」バリデーション・席タイプ選定に使う |
| 必要席タイプ×個数 | Varchar | ○ | 確保する枠（例：`table4 × 1`）。**物理卓番号ではない** |
| 事前メニュー有無 | Bool | ○ | あり=デシャップ予約ストックに載せる（C-2）。なし=席ロックのみ |
| 事前メニュー内容 | JSON | － | ありの場合の注文内容 |
| **割当卓番号** | Varchar | － | **予約時はnull → 来店時にホールが入力**。座席テーブルの座席IDと一致 |
| 状態 | Varchar | ○ | `booked`(枠確保) / `seated`(着席=卓確定) / `done`(完了) / `noshow` |

- **予約成立時**：予約枠テーブルに1行作る。`割当卓番号 = null`。物理座席テーブルはまだ触らない（or 仮ロックのみ）。
- **来店時**：ホールがA案で予約を選び卓番号入力 → 予約枠 `割当卓番号` を埋め `状態=seated`、座席テーブルの当該席を `occupied`＋`紐付け予約ID`セット。**この瞬間に予約⇔卓が一致**。

席構成：カウンター8 / 2名席3卓 / 4名席3卓（[要件定義（仮）.md](../要件定義（仮）.md) L292 座席マスタ）。
