# テーブルチェックAPI連携：取得したい情報の洗い出し（打合せ準備メモ）

> 目的：オーナーがテーブルチェック担当者と打合せする際に持ち込む「当店側が欲しいデータ」の一覧。
> これを基に、**どのAPI（プラン）を使うか・料金がいくらになるか**をテーブルチェック側と詰める。
> 対応：[Issue #74](../../../issues/74) テーブルチェックAPI連携の未確定事項（料金・データ形式・通知メール可否）。
> 出典：[deshup-spec.md §テーブルチェック連携](../deshup-spec.md)（48-53行目）／[db-design.md](../docs/db-design.md) §5.1 reservations・§9-8／[議事録 2026-06-18](2026-06-18_議事録_システム構成・メニュー.md) §4。

## 0. 前提（現状わかっていること）

- ✅ **API連携自体は可能**（テーブルチェック側から確認済み・6/18議事録＋コミット `6696ce4`）。ただし**追加料金が発生**。
- ❓ **金額・取得できるデータの中身と形式・予約通知メールの可否**の3点が未確定（Issue #74）。
- 📌 **現状は未接続**。連携コスト・仕様が確定するまでは KDS 側はデモデータで運用中（`deshup-spec.md`）。
- 📌 db-design.md では既に「テーブルチェックAPI取り込みに備えた形」で `reservations.external_id` / `source`（`manual` / `tablecheck`）を設計済み。**連携不可なら `source` は `manual` のみで運用**（保留事項#8）。

この文書は上記「中身の未確定」を打合せで潰すための**当店側の要求リスト**。

## 1. 当システムが欲しいデータ項目（優先度つき）

当店側の受け皿（KDS 予約ストックパネル・座席DB設計）が実際に使う項目ベースで整理。**Must＝これが無いと予約連携する意味が薄い／Nice-to-have＝無くても手動補完で回せる**。

### 1-1. Must（必須）

| 項目 | 用途 | 当店側の格納先 |
| --- | --- | --- |
| 予約ID（外部キーとして使える一意なID） | 重複取込防止（冪等）。テーブルチェック側の予約と当店DBの予約を紐付ける照合キー | `reservations.external_id` |
| 来店予定日時 | 予約ストックの時間順ソート／炊飯先行着手（来店30分前プロンプト）の起点 | `reservations.expected_at` |
| 人数（**大人・子供の内訳**） | 「大人1名1膳必須」バリデーション、予約カード表示 | `reservations.adults` / `kids` |
| 予約者名 | 来店時にホールが予約一覧から特定する際に使う | `reservations.customer_name` |
| 予約ステータス（確定／変更／キャンセル／ノーショー） | ストックからの除去・状態管理。変更検知ができないと二重表示や消し忘れが起きる | `reservations.status` |

### 1-2. Nice-to-have（あれば運用が楽になる）

| 項目 | 用途 | 当店側の格納先 |
| --- | --- | --- |
| 事前メニュー内容（メニュー名・数量・オプション） | 「メニュー予約」（炊飯先行着手の対象）か「席予約」（メニュー未定）かの判定・調理準備 | `reservation_items` |
| アレルギー情報 | 品目直下への常時表示（安全に関わる情報） | `reservation_items.allergies_text` |
| 連絡先（電話番号等） | ノーショー時の確認連絡 | 未設計（v2扱いでも可） |
| 希望席タイプ／人数枠（カウンター・2名卓・4名卓など） | 座席DBの `reservation_slots`（席タイプ×個数の枠確保）との対応 | `reservation_slots` |
| 予約変更・キャンセルの**リアルタイム通知**（Webhook等） | ポーリングだけだと変更に気づくまでのラグが生まれる | — |

> 事前メニュー（1-2の1番目）がテーブルチェック側で取得できない場合、「メニュー予約」という区分自体が成立しない可能性がある。**この可否は打合せの最重要確認ポイント**。

## 2. API仕様面の確認事項（料金に直結する部分）

Issue #74 の「データ形式」に対応。技術的な仕様が料金プランを左右するため、データ項目とセットで確認する。

| 確認事項 | 確認したい理由 |
| --- | --- |
| 取得方式：**Webhook（push）か REST API ポーリング（pull）か** | 現行KDSは1秒ポーリング設計（`window.KDS_ORDERS`）。テーブルチェック側もポーリングで足りるか、Webhook必須（＝別料金）かで実装方式が変わる |
| データフォーマット（JSON/CSV等）とスキーマの有無 | §1のMust項目がどのフィールド名・型で返るか。db-design.md の `reservations` テーブルへのマッピング表を作るために必要 |
| 更新頻度・反映までの遅延 | 直前キャンセル・当日予約への追従性に影響 |
| 認証方式（APIキー／OAuth等） | 実装コスト・鍵管理の手間に影響 |
| レート制限の有無 | ポーリング間隔の設計に影響 |
| テスト／サンドボックス環境の有無 | 本番データを使わずに実装・検証できるか |
| **料金体系**：初期費用／月額固定／従量課金（予約件数・APIコール数）／プラン別の取得可能データの差 | Issue #74 の核心。プランによって§1のMust/Nice-to-haveがどこまで取れるかが変わりうる |
| 既存のテーブルチェック契約（予約台帳）へのアドオンか、別契約・別料金か | 既契約との関係整理 |
| **予約通知メールの送信可否**（Issue #74 の3点目） | 来店時の確認連絡・ノーショー対策に使えるか |

## 3. 打合せのゴール（今回持ち帰りたい結論）

Issue #74 に書かれている3つの未確定事項を確定させる：

1. **API連携の追加料金**（初期費用・月額・従量、プラン別の金額）
2. **取得可能なデータの中身・形式**（§1のMust項目が全部取れるか。特に「大人/子供の人数内訳」「事前メニュー」の可否）
3. **予約通知メールの可否**

加えて、§2の技術仕様（Webhook/ポーリング・認証・更新頻度）も可能なら聞いておくと、実装側の設計（`db-design.md` の `reservations.source='tablecheck'` 経路）にそのまま反映できる。

---

## 4. 連携不可・想定より高額だった場合の代替

- `db-design.md` 保留事項#8のとおり、**テーブル設計自体は変えずに** `reservations.source` を `manual` のみで運用すればよい（設計は連携前提で作ってあるので手戻りは無い）。
- 運用は現行の「予約→オーナー/スタッフが手動で当店フォームに入力」を継続（`deshup-spec.md` の予約ストックパネル「＋追加ボタン」がその受け皿）。
- **§1のMust項目のうち一部だけ**（例：日時・人数・名前のみ、事前メニューは不可）が取れる場合は、部分連携＋残りは手動入力のハイブリッドも選択肢になる。
