## 4. 整理の提案（MUST / 切り離す）

> 「無駄なところを消していく」方針で、5/21 スコープに寄せ直す叩き台。**大城さんの判断を仰ぐ前提**。

| 区分 | 内容 | 根拠 |
|------|------|------|
| 🟢 **MUST（本体・残す）** | ①顧客タブレットでの注文 ②厨房へのオーダー表示（デシャップ） ③**こだわりの食べ方の伝達**（本題の core） | 5/21 スコープ＋§1 の本題 |
| 🟡 **要判断** | デシャップ端末を**産業用 → コンシューマー用＋防水ケース**に落とせるか／キッチンは**紙メイン＋補完で端末**でよいか | muku・女1号案。コスト激減 |
| 🔴 **切り離す候補** | 予約転記システム／座席管理システム／満席自動判定／店全体の機材見積もり | 大城さん「予約は切り離してOK」発言 |

### 補足：「API で引っ張る」案について（muku 案「APIひっぱてこればよくね」への検証）

**まず通信の"向き"を分けて考える。これを混同すると議論が噛み合わない。**

| 向き | 内容 | ローカルサーバーで可能か |
| --- | --- | --- |
| **push**：SaaS → ローカル | 予約が入ったら SaaS が店内に通知してくる（Webhook） | **❌ 無理**。SaaS が店内のローカルIPを叩く形になるが、ローカルは NAT/ファイアウォール内で**外から到達できない** |
| **pull**：ローカル → SaaS | 店内が定期的に SaaS の API を叩いて「新しい予約ある？」と取りに行く（ポーリング） | **△ 向きは通る**。発信元がローカル＝外向き通信は家庭用ルーターでも通る。SaaS は受け身で応答するだけ |

→ つまり **「SaaS からローカルを叩く」のは無理／「ローカルから SaaS を叩く」なら向きは通る**。muku 案は「向きが逆なら」成立しうる。

**だが、pull が技術的に通っても実用上は採れない（ここが本丸）：**

| 壁 | 中身 |
| --- | --- |
| ① そもそも API があるか | 中小向け予約SaaS は外部公開REST APIを**持たない／上位有料プランのみ**が多い。SaaS名すら未確定（議事録 §7）なので**今は確認不能** |
| ② pull できても満席は止まらない | pull は「入った予約を**取りに行く**」だけ。**満席判定＝予約を止める**には逆に SaaS の枠を**書き換える write API** が要り、ハードルが一段上がる |
| ③ 結局「転記が消える」止まり | pull が成立しても効果は「まりこさんの転記自動化」まで。§3 の本丸**「満席自動ストップ」は解決しない** |

→ 結論：**「API で引っ張る」は、向きを pull にすれば技術的には通るが、(a) SaaS が API を公開しているか不明 (b) 満席判定には無力、という二点で実用上は当てにできない**。
→ よって「APIで転記を楽にする(pull)」と「満席を自動で止める(write/連携)」は**完全に別問題**。前者すら API 次第、後者はサーバ構成（§1）とSaaSのwrite API次第で、いずれも**まず大城さんに SaaS の API 仕様を確認しない限り絵に描いた餅**。

## 5. 紙 vs モニターの論点（キッチン側）

- 現状クライアントは**紙でオーダー管理**しており、そこに**強い不満は無い**（「モニターにできるならしてみてもいい」程度）。
- 不満は「**紙がもったいない**」という点（muku）。
- → 割り切り案：「**お客さんが注文したら紙が自動で出てくる**だけ」でも本題（注文→厨房伝達）は満たせる。産業用モニターを必須にしなくてよい可能性。
- ⚠️ ただし、まりこさんは既に**モニター運用を想定して考えている**ため、いきなり「やっぱり紙で」は混乱を招く。**大城さん経由で意図をすり合わせてから**。
